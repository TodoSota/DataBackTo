# ゲーム概要

## 基本情報

|項目|内容|
|---|---|
|タイトル|DataBackTo|
|ジャンル|2D横スクロールアクション|
|プレイ人数|1人|
|対応機種|PC|
|使用エンジン|Unity|
|制作人数|2人|
|制作期間|2026年1月22日～|

---

## コンセプト

プレイヤーの状態（HP・コイン・ジャンプ回数など）を保存し、  
必要なタイミングで**過去の状態へ戻る**ことで攻略する  
リソース管理型アクションゲーム。

---

## ゲームシステム概要

※各項目に対応する画像やGIFを配置予定

1. **プレイヤーの状態を保存して「レシート」を発行**
2. レシートを使用すると**記録された状態へ復帰**
3. アクションを行うたびに**HPが消費**
4. ステージや敵から**コインを獲得**
5. コインを消費することで**高機動アクションを発動**

---

## 画面イメージ

以下のようなプレイ映像を掲載予定

1. 基本移動〜攻撃までの一連  
2. リソース消費アクション  
3. 状態保存 → 復帰システム  

---

## 遊び方・詳細説明

### 操作方法

#### 基本操作

| 行動 | 操作キー |
|---|---|
| 左右移動 | A / D または 矢印キー |
| ジャンプ | Space |
| 攻撃 | X / C / V / B / N / M |
| 下方攻撃（ヒップドロップ） | ジャンプ中に攻撃キー |
| 状態保存 | Enter長押し（2秒） |
| 状態復帰 | Enter |

---

#### リソース消費アクション

| 行動 | 操作キー | 仕様 |
|---|---|---|
| 多段ジャンプ | Shift + ジャンプ | 2段目以降で発動。現在段数分のコインを消費 |
| 射撃攻撃 | Shift + 攻撃 | 1コイン消費。コインを前方へ射出 |
| 強化ヒップドロップ | 空中でShift + 攻撃 | 10コイン消費 |

---

### リソース仕様

#### HP

主人公の体力。  
0になるとゲームオーバーとなる。

歩行やヒップドロップなど、体への負荷がかかるアクションを行うたびに徐々に減少する。

---

#### コイン

ステージ上や敵を倒すことで獲得できる。  
各種アクションの発動に使用する。

必要量を満たさない場合は、  
強化アクションは発動せず通常動作となる。

---

#### レシート

主人公の状態を記録する媒体。

記録される情報：

- HP
- コイン
- ジャンプ回数

レシートを使用すると、その状態に復帰する。

またHPが尽きた場合、レシートが残っていれば自動で消費されコンティニューとなる。  
レシートが無い状態でHPが0になるとゲームオーバー。

---

### 勝利条件・ゲームループ

ゲームは以下の画面で構成される。

- ステージ選択画面
- ステージ

プレイヤーはステージを選択して挑戦し、  
ゴール到達またはゲームオーバーでステージ選択画面へ戻る。

# 開発・設計について

## 制作目的

本開発は **TodoSota** と **otukebo-ya** の2名で制作した。

TodoSota 主導で行われたプロジェクトであり、 otukebo-ya は不足していた実装や技術部分について主に担当のため参加した。
ゲーム制作に際した TodoSota の主な目的は以下の通りである。

- **Unity未経験からの脱却**
- Unityを用いたゲーム制作経験の獲得
- 小規模チームでの制作工程の検証

これまでいくつかゲーム制作に関わった経験はあったが、  
プログラム面での開発工程については未知の部分が多かった。

そのため本プロジェクトでは

- Unityというゲームエンジンの使用感
- 実際の開発フロー
- 小規模チーム開発

を経験することを目的として制作を行った。

また副目的として、  
**少人数制作において失敗しにくい制作工程の検証**を行った。

---

## 制作工程設計

本プロジェクトでは、過去の制作経験から以下の問題が存在していた。

・最初から完成形を目指してしまい開発が進まなくなる  
・未確定部分の検討に時間を使い作業が止まる  
・途中で全体像を見失いモチベーションが低下する  

これらの問題を解決するため、制作工程を **2段階** に分け、以下の方針を設定した。

### 第一段階：ゲームの土台を完成させる

前半工程では

**「最適化ではなく変更耐性を優先する」**

という方針を採用した。

具体的には

- ゲームループの最小構成を定義
- その成立のみに集中して実装
- 未確定要素は一旦無視

という形で、**最速で動作するゲームの土台を構築**した。

ここで意図していたのは

> パフォーマンスの最大化ではなく  
> **変更に対する損失の最小化**

である。

未確定要素については、実際にゲームを動かす中で判断できるものと考え、  設計段階で時間をかけすぎないようにした。
主な取り組みは以下のとおりである。

 - 最小ゲームループの実装
 - 仮素材(キューブなど)によるステージの構築
 - プレースホルダ―を利用したUIの作成

---

### 第二段階：体験の拡張

前半で構築した土台に対して

- 仮置きしていたオブジェクト（キューブ・プレースホルダー）
- 簡易実装のUI
- 演出の無い処理

などを**本実装へ置き換える工程**を行った。

この段階では

- 見た目の改善
- フィードバックの追加
- 演出の追加

などを行い、ゲーム体験の完成度を高めていった。
